runtime

runtime.Gosched()

首先，main函数会创建一个协程，然后调用runtime.Gosched()，这个函数会切换到其他协程，然后打印hello。
如果没有runtime.Gosched()，那么main协程会很快的运行完毕，可能会出现一个world也打印不出来的情况，加上后必然先打印出一个world。

runtime.Goexit()

就当是这个goroutine运行完毕了，后续啥也不会干了

runtime.GOMAXPROCS()

GOMAXPROCS设置最大goroutine执行的cpu数量,很好理解，就是设置并发数

Mutex

常用的有互斥锁和读写锁
互斥锁是完全互斥，但是经常会有读多写少的需求，我们高并发的读取不涉及修改数据，那么就可以用读写锁

Once

Once保证只执行一次
线程安全，多个 goroutine 同时调用也不会有问题

atomic

由于mutex加锁要切换到内核态，而且涉及到上下文切换，性能会下降
而atomic是用户态的，基于 CPU 提供的原子指令，对于常见的数据类型使用atomic操作性能更好

Worker Pool

Worker Pool（工作池）是一种并发设计模式，它维护一组预先创建的 goroutine（工人），这些 goroutine 等待执行任务。这种模式可以控制并发数量，避免创建过多的 goroutine，提高系统性能。

↓ ↓ ↓
工人1 工人2 工人3
↓ ↓ ↓
————- results 队列 ——-

练习

练习题目：生产者-消费者模式
题目要求：
编写一个程序实现生产者-消费者模式：
创建一个生产者 goroutine，生成 50 个随机数（范围 1-100）
创建 3 个消费者 goroutine，从同一个 channel 接收并处理数据
每个消费者接收到数据后，打印出处理结果（例如：消费者ID + 接收到的数字）
主程序需要等待所有生产者和消费者完成工作后，打印”所有任务完成”
具体要求：
使用 channel 在生产者和消费者之间传递数据
生产者完成生产后要关闭 channel
消费者使用 range 从 channel 读取数据，自动感知 channel 关闭
使用 sync.WaitGroup 等待所有 goroutine 完成

Context

Context 是 Go 语言标准库提供的一种上下文传递机制，用于在多个 goroutine 之间传递取消信号。Context 可以在任意层级传递，并且可以取消，也可以超时。
Context 的主要作用是：
取消（cancel）：上游通知下游“立即停止”。
超时（timeout/deadline）：到点自动取消。
传值（value）：跨 API 边界携带请求级公共数据（如 trace-id、用户身份），避免每个函数都加参数,这在项目开发时候特别有用。
